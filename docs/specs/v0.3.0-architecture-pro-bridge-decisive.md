# ADR: v0.3.0 architecture — JobScheduler content-URI trigger + catch-up (FGS 完全削除)

- **Date**: 2026-05-07
- **Status**: Accepted
- **Decision Maker**: yamaga101 + L4 council (Opus + Gemini 3.1 Pro + GPT-5.5) + GPT-5.5 Pro Bridge
- **Supersedes**: v0.2.0 の "always-on FGS + ContentObserver" 構成

## Context

v0.2.0 出荷直後 (2026-05-06)、Samsung S25 Ultra (Android 15 / One UI 7 / `targetSdk=35`) における BG 死亡 + 診断逸失の改善を v0.3.0 として議論した。

L4 council 起動時の「現実主義 catch-up」採用方針:

- BG 死を諦めて起動時 catch-up 主軸
- Persistent log + heartbeat (Room rolling)
- PeriodicWork は入れない

3 体合意の core 設計:

1. cursor `lastSeenId` を DataStore persist, `MainViewModel.init`+`onResume` で idempotent catch-up
2. 初回ガード (`currentMaxId < persistedId` 等で baseline reset)
3. heartbeat (FGS ticker + Worker doWork)
4. Persistent log Room
5. UX StatusBanner 1 行
6. PeriodicWork 入れない
7. Battery / Samsung 既実装の補足のみ

ただし **Codex (GPT-5.5) のみが致命 Warning を提示**:

> Android 15 / `targetSdk=35` で `dataSync` FGS は **24h 累積 6h cap**。`BOOT_COMPLETED` から `dataSync` FGS **起動不可**。

3 体合意の枠から外れたため、**GPT-5.5 Pro Bridge** (operating.md §3 Consensus Risk Gate) で独立検証。Pro Bridge が **architecture-level inversion** を発見:

> 常時 FGS は ContentObserver の前提条件ではない。Android 7.0 で `ACTION_NEW_PICTURE` 廃止と同時に提供された official replacement = **`JobScheduler.addTriggerContentUri(MediaStore.Images.Media.EXTERNAL_CONTENT_URI, FLAG_NOTIFY_FOR_DESCENDANTS)`**。MediaStore mutation で wake する event-driven trigger で常時 process 不要。

### Codex Warning の事実認定 (Pro Bridge による)

| 主張 | 真偽 | 根拠 |
|------|------|------|
| Android 15 `targetSdk=35` で `dataSync` FGS 24h 累積 6h cap | ✅ 真 | `Service.onTimeout()` → grace period → `RemoteServiceException` ([Android 15 behavior changes](https://developer.android.com/about/versions/15/behavior-changes-15)) |
| `BOOT_COMPLETED` から `dataSync` FGS 起動不可 | ✅ 真 | `FGS_BOOT_COMPLETED_RESTRICTIONS` compat flag が target-35 で default ON。`dataSync` は exception list (`location/connectedDevice/remoteMessaging/health/systemExempted/specialUse`) に含まれず → `ForegroundServiceStartNotAllowedException` ([Android 15 FGS types](https://developer.android.com/about/versions/15/changes/foreground-service-types)) |
| 24h+ 連続稼働で発生する例外 | `RemoteServiceException` (not `ForegroundServiceDidNotStartInTimeException`) | 後者は `startForegroundService` 後 `startForeground` を時間内に呼ばなかった場合 |

**現状 v0.2.0 が S25 で動いている理由**: Samsung が 6h 前に kill / 前面復帰で timer reset / 累積未達、のいずれか。**24h+ 連続稼働で時限爆弾**。

## Decision

v0.3.0 の architecture を以下に変更する:

```
BootReceiver / ACTION_MY_PACKAGE_REPLACED / MainActivity#onResume
  ├─ scheduleMediaStoreTriggerJob() (idempotent)
  └─ enqueueUniqueWork("catchup", CatchUpWorker)

MediaStoreTriggerJobService.onStartJob(params)
  ├─ enqueueUniqueWork("catchup", CatchUpWorker)
  ├─ scheduleMediaStoreTriggerJob() (reschedule before finish)
  └─ jobFinished(params, false)

CatchUpWorker
  ├─ scan MediaStore _ID > persistedLastSeenId
  ├─ filter Screenshots / DCIM/Camera
  ├─ baseline guard
  ├─ enqueue UploadWorker per new image
  └─ update cursor / Room log / status DataStore
```

### 採用要素

1. **`UploadForegroundService` 完全削除**
   - manifest からも declaration 削除
   - `BootReceiver` から FGS start 経路削除
   - `MainViewModel.ensureAutoSyncRunning` 削除
   - `FOREGROUND_SERVICE` / `FOREGROUND_SERVICE_DATA_SYNC` permission 削除
   - persistent notification 廃止
2. **`MediaStoreTriggerJobService`** (JobService) で content URI trigger
   - `setPeriodic` / `setPersisted` 不可、boot/package/app-open で reschedule 必須
   - `getTriggeredContentUris()` は信頼しない (50 URI 超過時 null)
3. **`CatchUpWorker`** (CoroutineWorker) で scan + enqueue
   - cursor (`lastSeenId`) は DataStore persist
   - `cursor_initialized=false` または `currentMaxId < persistedId` (factory reset / app reinstall) → baseline reset
   - dedupe: `enqueueUniqueWork("shot-sync-upload-id-$id", KEEP)` で重複防止
   - 手動 upload も同 unique key 使用 (旧 v0.2.0 の `manual-id-$id` を統一)
4. **Room Persistent Log** (rolling 500件 / 7日)
   - `DiagnosticsLog` を hot in-memory facade として残し、Room async mirror
5. **Status Banner** (上部 1 行)
   - `Last scan: 42 min ago · Last trigger: 51 min ago · Pending: 3`
   - warn 閾値: `last_trigger > 6h`
6. **Samsung 設定手順カード**
   - App info → Battery → Unrestricted
   - Deep sleep apps → exclude
   - Permission 既実装の `requestIgnoreBatteryOptimizationsIntent` をボタン化

### 維持事項

- `targetSdk=35` 維持 (downgrade は dodge にすぎない、Doze/App Standby は無関係に効く)
- `MediaStoreScreenshotObserver` も削除 (常時 process 不要のため不要)
- `BootReceiver` は残す (Job 再スケジュール + catchup enqueue 用途、FGS 起動はしない)

## Consequences

### Positive

- **Android 15 target-35 での legality 保証**: `RemoteServiceException` / `ForegroundServiceStartNotAllowedException` リスク完全消失
- **コード簡素化最大**: 常時 process / heartbeat ticker / persistent notification の運用負荷消失
- **Honest contract**: 「instant upload を約束しない、near-real-time best-effort + guaranteed catch-up on app open」と明示
- **Samsung deep sleep に対し対称**: PeriodicWork も FGS も同じく落とされる、JobScheduler trigger は MediaStore mutation 駆動なので「死んでる時は等しく死ぬ、生きてる時は等しく拾う」

### Negative

- **Persistent notification 消失**: 「監視中」の visual confirmation がなくなる → Status Banner で代替
- **Trigger reschedule 漏れリスク**: boot/package/app-open のいずれかが取れないと永続的に dead → `MainActivity.onResume` での re-scheduling が安全網
- **`getTriggeredContentUris()` 信頼不可**: 50 URI 超過時 null、deadline trigger 時 null。「something changed; run scanner」として扱う
- **Doze / App Standby Restricted bucket では trigger 遅延・cancel**: これは PeriodicWork も同じで現実主義の前提

### Neutral

- 即時性は OEM に依存。Samsung deep sleep 中はアプリ起動まで catch-up されない
- `ACTION_USER_PRESENT` / `ACTION_SCREEN_ON` も将来 trigger 候補だが、Samsung deep sleep 配下では receiver も inert になるため v0.3.0 では採用しない

## Alternatives Rejected

### Alternative 1: 3 体合意通り「FGS-best-effort + catch-up」(council 当初案)

**却下理由**: Android 15 target-35 で 24h+ 連続稼働すると `RemoteServiceException` で crash。`BOOT_COMPLETED` → `dataSync` FGS は `ForegroundServiceStartNotAllowedException` で起動不可。**時限爆弾**を抱える。

### Alternative 2: 純粋 PeriodicWork 15 min + on-launch catch-up (Opus inversion 案)

**却下理由**: WorkManager は JobScheduler 下層、Doze で動かない。App Standby Restricted bucket では 1 日 1 回程度。「15 min 毎 scheduled」と log されるが実態不定 → **診断混乱の typifier**。content-URI trigger の方が semantically aligned (polling でなく event-driven)。

### Alternative 3: `targetSdk=34` ダウングレード (Opus inversion 案)

**却下理由**: 単なる dodge。Doze / App Standby は target SDK 無関係に効く。**v0.2.1 hotfix の escape hatch としてのみ許容**、v0.3.0 design 基盤にしない。Android 16 で結局 forced。

### Alternative 4: `ACTION_USER_PRESENT` / `ACTION_SCREEN_ON` broadcast trigger (Opus inversion 案)

**却下理由**:
- `ACTION_USER_PRESENT` は Android 8+ 暗黙 broadcast 制限の **exemption list に含まれない** (Pro Bridge official confirm)
- `ACTION_SCREEN_ON/OFF` は manifest 不可、`registerReceiver()` のみ
- Samsung deep sleep では receiver 自体 inert
- bootstrap 用途に留める

### Alternative 5: FGS を "Live mode" として残す

**却下理由 (v0.3.0 では)**: scope 拡大。まず JobScheduler trigger 単独の挙動を実機検証してから、必要なら v0.3.x で追加検討。
